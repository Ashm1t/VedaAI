# Journey.md

This is the working log for Libra's shift from an assignment generator into an agentic LaTeX document workspace. I am writing this as an Applied AI Engineer would: not just "what changed", but why it changed, what broke or could break, what fallback exists, and how a non-specialist can picture the system.

## Current Product Direction

Libra is now centered on one core experience: an agentic chat interface that helps users create, edit, inspect, and export structured documents. The old assignment generator pipeline is no longer the product center. It can stay as legacy reference material for now, but new engineering effort should serve the agent workspace.

Plain-language version: Libra is no longer a vending machine for assignments. It is becoming a document co-pilot with a workshop, memory, tools, and a PDF preview table.

## Why The Agent Needed A New Shape

The early agent could talk, generate LaTeX, compile, and show a PDF. That proved the skeleton worked. The problem was that it behaved too much like a single loose assistant. It could forget formatting constraints, generate weak structure, ignore templates, or produce LaTeX that technically compiles but looks bad.

The fix is not "make the prompt bigger". Bigger prompts become expensive, confusing, and harder to debug. The better fix is to split the work into smaller responsible parts:

- One part understands the user's files and source material.
- One part searches memory and chooses the best template or formatting pattern.
- One part writes and repairs the LaTeX.
- One coordinator decides the order of operations and keeps the state clean.

Layman comparison: instead of asking one person to be librarian, designer, typist, printer technician, and project manager at once, we give each job to a specialist and let a project manager coordinate them.

## Recommended Runtime Stack

The recommended stack is:

- **LangGraph** for orchestration.
- **LangChain** for tool calling.
- **LangSmith** for tracing, debugging, evaluation, and monitoring.

Why this stack fits:

- LangGraph gives us a stateful workflow. Libra needs a clear path like `Intent -> Ingest -> Retrieve -> Clarify -> Build -> Compile -> Repair -> Export`.
- LangChain gives us a standard way to expose tools like `extract_pdf`, `search_templates`, `compile_latex`, and `call_mcp_tool`.
- LangSmith gives us a flight recorder. When the generated PDF is wrong, we can inspect exactly which prompt, retrieval result, tool call, compile error, or repair step caused it.

Layman comparison: LangGraph is the railway map, LangChain tools are the machines at each station, and LangSmith is the CCTV/control-room log that shows where the train got delayed.

Official docs checked while making this recommendation:

- LangGraph is described by LangChain as a low-level orchestration runtime for long-running, stateful agents with durable execution, streaming, and human-in-the-loop support.
- LangChain tools are callable functions with defined inputs and outputs that agents can invoke when they need external capabilities.
- LangSmith traces record model calls, tool calls, decision points, and full agent execution paths for debugging and monitoring.

## Multi-Agent Design

The proposed architecture has three specialist agents plus one deterministic coordinator.

### 1. IngestionAgent

Job: read the world coming into Libra.

Inputs:

- PDF files.
- DOCX files.
- Images.
- Markdown.
- LaTeX source.
- Existing Libra artifacts.

Outputs:

- Extracted text.
- Recovered `.tex` or `.bib` source if present.
- Document outline.
- Field map such as title, author, dates, sections, citations, contact details, or assignment metadata.
- Confidence level.
- Artifact ids for large extracted content.

Fallbacks:

- If a PDF was generated by Libra, load the original source from the artifact store.
- If the PDF has embedded source files, extract those.
- If the PDF has selectable text, extract and structure that.
- If it is scanned, use OCR.
- If OCR is weak, ask the user for clarification rather than pretending certainty.

Layman comparison: this agent is the clerk who opens the envelope, sorts the papers, labels them, and tells the rest of the team what is inside.

### 2. RetrievalAgent

Job: search the RAG memory and template catalog.

Inputs:

- User intent.
- Mode.
- Ingestion summary.
- Required document type.
- Known fields.

Outputs:

- Shortlist of candidate templates.
- Matching reasons.
- Required missing fields.
- Renderer adapter id.
- Compile profile id.

Fallbacks:

- If no strong template match exists, use sandbox mode and ask the user whether to continue without a known template.
- If multiple templates tie, show compact template cards instead of choosing silently.
- If a template has poor compile health, demote it even if semantic similarity is high.

Layman comparison: this agent is the librarian. It should not write the book. It finds the right shelf, pulls the best few books, and explains why they fit.

### 3. LatexBuilderAgent

Job: produce and repair the actual document source.

Inputs:

- Selected template id.
- Renderer adapter.
- Field map.
- User instructions.
- Source artifacts.
- Retrieval snippets.

Outputs:

- `.tex` source.
- Optional `.bib` source.
- Assets.
- Human-readable summary.
- Compile-ready artifact ids.

Fallbacks:

- If required fields are missing, ask targeted questions before generating.
- If compile fails, run a bounded repair loop.
- If formatting breaks badly, revert to the last compile-safe source snapshot.
- If the selected template is incompatible with available engines/packages, ask RetrievalAgent for the next candidate.

Layman comparison: this agent is the typesetter. It should not wander the library forever; it should use the chosen template and carefully place content into it.

### 4. DocumentCoordinator

Job: control the workflow.

This should be implemented as a LangGraph graph, not as a fourth creative agent. The coordinator should route, checkpoint, store state, enforce budgets, and decide whether the next step is ingestion, retrieval, clarification, build, compile, repair, or export.

Fallbacks:

- If intent is unclear, stay conversational and ask a simple question.
- If the user is just chatting, do not trigger the full pipeline.
- If a tool fails, report the failing step and retry only where safe.
- If an external MCP tool wants to mutate files or call outside services, ask permission.

Layman comparison: this is the conductor. The conductor does not play every instrument; it keeps the orchestra synchronized.

## PDF Ingestion And Source Recovery

Direct PDF text editing is not realistic. PDFs are final render artifacts, more like printed paper than editable drafts. So Libra should treat PDFs as clues that help recover or reconstruct editable source.

Best path:

1. Check if the PDF belongs to a Libra session and load the original `.tex`.
2. Check for embedded files inside the PDF.
3. Extract selectable text.
4. Detect LaTeX fragments or structure markers.
5. Reconstruct a document outline.
6. OCR only if text extraction fails.
7. Store confidence and provenance for every extracted item.

Why this matters:

- Real source can be edited precisely.
- Reconstructed source should be handled carefully.
- OCR output needs verification.
- The user should know whether the agent is editing real source or rebuilding from a rendered PDF.

Fallback:

- If source cannot be recovered, switch to "rebuild from PDF" mode and ask the user whether preserving visual layout or preserving content matters more.

## RAG Memory Improvements

The current memory should evolve from "chunks of template text" into "document intelligence".

Better memory types:

- Template metadata.
- Section schemas.
- Required and optional fields.
- Compile requirements.
- Known formatting failure modes.
- Rendered examples.
- User/project preferences.
- Successful previous generations.
- Repair recipes.

Template selection should not be pure semantic search. It should combine:

- Semantic similarity.
- Mode match.
- Required field coverage.
- Compile health.
- Renderer adapter fit.
- Recent success.
- Missing dependency penalties.

Layman comparison: choosing a template should be like choosing a recipe. The agent should know not just the recipe name, but ingredients, cooking time, tools needed, common mistakes, and whether it has worked before.

## Tool Layer And MCP Plan

Libra needs a proper tool layer before MCP is exposed broadly.

Internal tools should include:

- `extract_pdf`
- `extract_pdf_attachments`
- `extract_docx`
- `ocr_image`
- `search_templates`
- `get_template_source`
- `compile_latex`
- `validate_latex`
- `autofix_latex`
- `read_source`
- `write_source`
- `diff_source`
- `export_project_bundle`

Each tool should have:

- A name.
- A description.
- A typed input schema.
- A typed output shape.
- A permission level.
- A read-only or mutating flag.
- A token cost hint.
- A compact UI activity event.

MCP should be introduced behind a backend service:

```text
server/src/services/mcp/
  McpConnectionManager.ts
  McpServerRegistry.ts
  McpToolAdapter.ts
  McpPermissionPolicy.ts
  mcpConfig.ts
```

Fallbacks:

- Start with read-only MCP tools.
- Require permission for mutating MCP tools.
- If an MCP server is unavailable, show it as disconnected and continue with local tools.
- Store large MCP results as artifacts and summarize them for the agent.

Layman comparison: MCP is like giving Libra a plugboard for external equipment. The tool layer is the safety panel that decides which plugs are allowed, which need permission, and which results should be stored instead of shoved into the chat.

## UI Direction

The agent screen should become calmer and more capable:

- Chat remains the primary surface.
- Template cards appear only when intent calls for them.
- Clarification fields appear only when needed.
- Source editor is toggleable, not always visible.
- PDF preview remains prominent.
- Tool calls appear as compact activity logs, not giant panels.
- Artifact history should make it easy to roll back.

Fallback:

- If the agent cannot confidently proceed, it should ask one or two precise questions instead of opening a big form.

## First Implementation Step Completed

I added a safe backend contract layer without changing live `/agent` behavior yet.

Files added:

- `server/src/services/agentOrchestrationContracts.ts`
- `server/src/services/agentToolRegistry.ts`
- `server/src/services/mcp/mcpContracts.ts`
- `server/src/services/mcp/mcpConfig.ts`
- `server/src/services/agentInternalTools.ts`
- `server/src/services/agentDocumentGraph.ts`
- `server/src/services/agentObservability.ts`

Why these were added:

- `agentOrchestrationContracts.ts` defines the future LangGraph-shaped state, stages, ingestion packet, retrieval shortlist, build request, and build result. This gives the project a shared vocabulary before wiring in LangGraph.
- `agentToolRegistry.ts` defines a small internal tool registry. This is the stepping stone toward LangChain tools without forcing a dependency migration in the same edit.
- `mcpContracts.ts` defines what MCP servers, tools, resources, calls, and permission decisions look like.
- `mcpConfig.ts` loads `.libra/mcp.json` safely and normalizes MCP server definitions.
- `agentInternalTools.ts` registers current backend capabilities as agent tools: template search, template source loading, LaTeX validation, deterministic autofix, and PDF compilation.
- `agentDocumentGraph.ts` creates the first LangGraph skeleton with the future document stages: intent, ingestion, retrieval, clarification, build, compile, repair, and export.
- `agentObservability.ts` centralizes LangSmith runnable tags and metadata so graph/tool traces stay searchable.

Dependencies added to the server:

- `@langchain/langgraph`
- `langchain`
- `langsmith`
- `zod`

Environment documentation updated:

- `.env.example` now includes `LANGSMITH_TRACING`, `LANGSMITH_API_KEY`, and `LANGSMITH_PROJECT`.

Why this was the right first step:

- It does not break the currently working agent.
- It creates clean seams for LangGraph, LangChain, and MCP.
- It lets Claude/Codex continue from typed interfaces instead of trying to infer architecture from scattered prose.
- It keeps the assignment-generator removal and agent runtime migration decoupled.

Validation:

- `npm --prefix server run build` passed.
- After adding LangGraph, LangChain, LangSmith, tool wrappers, graph skeleton, and observability helper, `npm --prefix server run build` still passed.

Fallback:

- If LangGraph migration takes longer than expected, the existing `agentService.ts` can keep running while new graph nodes are implemented behind these contracts.

## Important Engineering Principle

Do not make the agent smarter by making every prompt enormous.

Make it smarter by:

- Giving each step the right context.
- Passing artifact ids instead of giant blobs.
- Keeping tools typed.
- Keeping graph state explicit.
- Tracing every decision.
- Asking the user for missing information at the right time.

This is the difference between an impressive demo and a reliable document engineering system.
